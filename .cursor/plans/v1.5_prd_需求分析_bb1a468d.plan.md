# 0755@Hermes Desktop v1.5 PRD 需求分析报告

## 一、PRD 核心定位

v1.5 要把 Hermes Desktop 从 **"Chatbox + Agent 配置壳"** 升级为 **"Portal Desktop 统一入口"**。核心变化：

1. Electron 不再只管 Hermes Agent，还要管 `ai-os-full`（Next.js + Express + PostgreSQL）本地全栈服务
2. 默认首页从 Chat 改为 Portal Home（通过 WebContentsView 挂载本地 Next.js 页面）
3. 新增 Runtime Guard 机制——Gateway 未运行时阻断进入 Portal
4. 新增 `AiOsRuntimeSupervisor` 管理 ai-os-full 的 backend/frontend 进程
5. 新增 `RuntimeStateRepository` 统一持久化所有服务状态

## 三、关键技术决策分析

### 决策 1：runtime_services 表 vs 扩展 profile_runtimes 表

PRD 提议新建 `runtime_services` + `runtime_events` + `runtime_settings` 三张表。但现有 `profile-runtime-db.ts` 已有 9 张表，其中 `profile_runtimes` 和 `runtime_events` 结构高度相似。

**建议**：不另起炉灶，而是在现有 `~/.hermes/desktop/profile-runtime.db` 中新增一张通用 `runtime_services` 表，用 `service_type` 区分 `hermes-gateway` / `aios-backend` / `aios-frontend` / `postgres`。现有 `profile_runtimes` 表保留给多 Profile Gateway 使用，`runtime_events` 表则可直接共用（加 `service_id` 字段做区分）。

**风险**：如果 PRD 坚持独立 DB 文件 `desktop-runtime.db`，则需要双 DB 管理开销。

### 决策 2：WebContentsView 管理——复用 BrowserController 还是独立

PRD 明确 Portal Web App 用独立 WebContentsView，且需要 reload / devtools / session partition。这与 `BrowserController` 的能力完全一致，但两者的使用场景不同：

- `BrowserController` 是为 Web Operator 设计的，支持多个 view、自动化操作、DOM 快照、截图
- Portal WebContentsView 是"主工作区"，只需一个固定 view，不需要自动化能力

**建议**：新建 `AiOsWebContentsController`，不继承 `BrowserController`，而是提取公共的 view 创建/bounds 管理为工具函数。这样保持职责清晰，避免 Portal 页面误触 Web Operator 自动化逻辑。

### 决策 3：App 路由变更的影响范围

当前 `App.tsx` 的生命周期是 `splash → welcome → installing → setup → main(Chat)`。PRD 要求改为 `main(Portal Home)`。

影响面：

- [src/renderer/src/App.tsx](src/renderer/src/App.tsx) — 终态跳转目标
- [src/renderer/src/components/layout/Layout.tsx](src/renderer/src/components/layout/Layout.tsx) — 默认 view 从 Chat 改为 AIOSHome
- 侧边栏导航顺序调整
- 需要新增 Portal Home 前的 Gateway 状态检查（Runtime Guard 逻辑）

**建议**：在 Layout 层增加 Gateway 状态订阅，当 status !== 'running' 时拦截 Portal Home 渲染，显示 RuntimeGuard 组件。

### 决策 4：ai-os-full 进程管理方式

PRD 区分了开发阶段（`pnpm dev`）和生产阶段（分别启动 backend / frontend）。

**建议分两步**：

- Batch 1-2：用 `child_process.spawn` 分别启动 `@portal/server` 和 `@portal/web`，与现有 `hermes-local-adapter.ts` 的 spawn 模式一致
- 后续：增加 `pnpm --filter` 精确控制和 production build 启动

**风险**：ai-os-full 依赖 pnpm、Node.js 18+，Windows 环境下需要确保这些工具已安装。PRD 的 runtime 目录规划了内置 `node/` 和 `pnpm/`，但 P0 阶段可能不含打包能力，需先假设用户机器已有 Node.js + pnpm。

## 四、PRD 中的潜在问题与建议

### 4.1 端口冲突风险

PRD 规划了 5 个服务端口（8642/8787/8000/3000/55432），加上现有 7 个 Profile Gateway 端口（8642-8648），共 12+ 端口。

**建议**：

- 复用现有 `port_allocations` 表统一管理
- Portal 端口在 `runtime_settings` 中可配置
- 启动前全部做端口检测（FR-009）

### 4.2 PostgreSQL 强依赖

PRD 后端需要 PostgreSQL，但 P0 未含本地 PG 方案（P1 FR-103 才考虑 portable PG）。

**风险**：这意味着 P0 版本要求用户预先有 PostgreSQL 可用——对于"内部电脑桌面安装"场景，这是一个阻塞点。

**建议**：P0 增加 PostgreSQL 连接检测 + 连接配置 UI，并在 Runtime Guard 中显示 PG 状态。可考虑使用 SQLite 替代方案或 embedded PG（pg-embedded）作为临时 fallback。

### 4.3 环境变量安全

PRD 要求生成 JWT_SECRET、ENCRYPTION_KEY 等敏感值并写入 `.env.desktop.local`。

**建议**：

- 使用 `crypto.randomBytes()` 生成
- `.env` 文件权限限制为当前用户
- 生成一次后持久化到 `runtime_settings`，避免重复生成导致 JWT 失效
- 不在 Renderer 暴露这些值

### 4.4 NSIS 配置改动

PRD 要求从 `oneClick: true` 改为 `oneClick: false` + `allowToChangeInstallationDirectory: true`。

**影响**：这是破坏性变更——现有用户升级时安装体验会改变。建议作为独立 PR 处理，并充分测试升级路径。

## 七、PRD 遗漏 / 需确认项

1. **ai-os-full 仓库访问方式**：PRD 说"私有 Git"，但未指定 Git URL、认证方式（SSH / HTTPS token）、tag/branch 策略
2. **Node.js / pnpm 来源**：P0 是否假设用户已安装？还是需要像 Python 一样做 bootstrap？
3. **PostgreSQL 连接信息来源**：谁提供 DATABASE_URL？是 Settings UI 手动填写，还是有默认值？
4. **ai-os-full 的 db:migrate 何时运行**：每次启动前？还是只在安装/更新时？
5. **Hermes WebUI (8787) 是否 P0**：PRD 环境变量含 `HERMES_WEBUI_BASE_URL`，但 P0 功能清单未提及 WebUI 管理
6. **Portal Home 语言路由**：`/zh` 是硬编码还是跟随 Desktop i18n 设置？
7. **profile-runtime-db vs 新 desktop-runtime.db**：需明确是共用还是独立
