## 1. 背景

v7.4.2 已经恢复 Chat 主入口，并在 ComposerBar 增加 Work Controls，用于选择 Expert、Skill、Permission，并在选择 Expert + Skill 后调用 MCP Expert Gateway。

当前本地测试发现：

```text
1. Expert 下拉菜单向下展开。
2. Skill 下拉菜单向下展开。
3. Permission 下拉菜单向下展开。
4. 菜单会挤占 Composer 区域。
5. 菜单容易被窗口底部、任务栏或 composer container 裁剪。
6. 效果与 WorkBuddy 的 Composer 控件菜单不一致。
```

WorkBuddy 的交互方式是：

```text
Composer 底部控件点击后，菜单在控件上方浮出。
```

因此 v7.4.3 只修复 Work Controls 的 Popover 显示方式。

## 4. 当前问题定位

当前 `WorkPopoverSelect` 的行为是：

```text
1. open=true 时在当前组件内部渲染菜单。
2. 菜单跟随 DOM 流布局。
3. 菜单默认在按钮下方。
4. 缺少 placement 配置。
5. 缺少 viewport 边界计算。
6. 缺少 portal。
7. outside click 只判断 rootRef，不支持 portal panel。
```

这会导致：

```text
1. Composer 底部空间不足时，菜单被遮挡。
2. 父容器 overflow 可能裁剪菜单。
3. 下拉菜单不符合 WorkBuddy 的向上浮出体验。
4. 后续 Expert / Skill 列表较长时可用性变差。
```

## 8. 定位算法

### 8.1 Top placement

```ts
const gap = 8;
const rect = trigger.getBoundingClientRect();
const width = menuWidth ?? Math.max(rect.width, 280);

const availableTop = Math.max(rect.top - gap - 8, 160);
const height = Math.min(maxMenuHeight, availableTop);

floatingRect = {
  left: rect.left,
  top: Math.max(8, rect.top - height - gap),
  width,
  maxHeight: height,
};
```

### 8.2 Bottom placement

```ts
const gap = 8;
const rect = trigger.getBoundingClientRect();
const viewportHeight = window.innerHeight;
const width = menuWidth ?? Math.max(rect.width, 280);

const availableBottom = viewportHeight - rect.bottom - gap - 8;
const height = Math.min(maxMenuHeight, Math.max(160, availableBottom));

floatingRect = {
  left: rect.left,
  top: rect.bottom + gap,
  width,
  maxHeight: height,
};
```

### 8.3 重新定位事件

```text
1. open 时立即定位。
2. window resize 时重新定位。
3. window scroll 时重新定位，capture=true。
4. options.length 变化时重新定位。
```
