# PRD v1.4：Work 任务窗口与实时 Agent SSE 交互方案

版本：v1.4
产品线：Copilot Desktop / Work 工作台
主题：任务 / 任务窗口 / Layout UI / 实时 Agent Event Stream
实施对象：ai-os-desktop / copilot-desktop
实施工具：Cursor / Spec Driver
核心目标：将 Work 工作台的“任务窗口”建设为用户核心操作平面，替代原先 Chat / Run / Artifact 分散交互，并支持专家、专家团队、技能、MCP、Web 上下文以实时 SSE 事件流方式进入同一个任务窗口。

---

## 1. 背景

当前 Copilot Desktop 已经完成第一轮 Expert MCP 调用与 Work 工作台结构优化。产品概念已重新定义：

```text id="7l4kxp"
Copilot Desktop
  ├─ Work 工作台
  │   ├─ 任务
  │   │   └─ 任务窗口：核心操作平面
  │   ├─ 专家
  │   ├─ 专家团队
  │   ├─ 技能 / MCP 能力
  │   └─ 运行时与治理
  │
  └─ Web 工作台
      ├─ Web Browser
      ├─ Hermes Panel / Web Task Panel
      └─ 技能
```

v1.4 的核心是把 `任务窗口` 做成用户每天使用的主界面，而不是继续以“对话页、专家运行页、成果中心”分散承载任务。

---

## 2. 产品定位

### 2.1 任务窗口定义

```text id="qsczfa"
任务窗口是 Copilot Desktop Work 工作台的核心操作平面。

它承载一个任务从目标输入、专家/团队选择、技能调用、上下文引用、实时执行、协作分派、工具调用、权限确认、产物生成、结果预览到持续追问的完整生命周期。
```

### 2.2 核心用户路径

```text id="2id10x"
打开 Work 工作台
  → 新建任务
  → 输入任务目标
  → 选择专家团队 / 专家 / 技能 / 应用 / 权限
  → 提交任务
  → 任务窗口实时展示 Agent 流
  → 展示团队成员分派与执行状态
  → 展示工具调用与审批确认
  → 生成任务输出
  → 右侧预览报告 / 文件
  → 用户继续追问或追加执行
```

### 2.3 产品原则

```text id="flwsgi"
1. WorkBuddy 只作为参考产品能力，不作为产品命名。
2. 本地 Hermes 统一改为 Work 工作台。
3. 会话统一改为任务。
4. Runs / Artifacts 统一收敛为 Task Event / Task Output。
5. Expert MCP 不再按异步任务 + 成果中心交互，改为实时 Agent SSE Event。
6. 任务窗口承载所有运行过程，不再单独突出“专家运行”和“成果中心”一级入口。
7. 专家、专家团队、技能、MCP、Web 上下文全部服务于任务窗口。
```

## 3. v1.4 建设目标

### 3.1 产品目标

```text id="wkx60p"
1. 建设 Work 任务页，作为 Work 工作台默认首页。
2. 建设任务窗口三栏 Layout。
3. 建设任务首页输入区，支持专家团队、模式、技能、应用、权限选择。
4. 建设 Agent 实时事件流渲染区。
5. 建设专家团队协作展示能力。
6. 建设任务右侧输出预览面板。
7. 建设任务上下文、参与者、技能、治理面板。
8. 支持从专家页 / 专家团队页 / Web 工作台创建或绑定任务。
9. 支持任务持续追问、停止、继续、重新生成。
```

### 3.2 工程目标

```text id="dk3mgt"
1. 不破坏现有 Electron Main / Preload / Renderer 分层。
2. 不让 Renderer 直接访问 nodeskclaw、token、本地文件系统。
3. 不在 UI 组件中直接调用 window.hermesExperts / window.workspaceChat。
4. 建立 workApi 作为 Renderer 语义适配层。
5. 建立 WorkTask / WorkTaskEvent / WorkOutput 等统一模型。
6. 建立 TaskWindow 组件分层。
7. 建立 SSE 事件归一化与前端 event store。
8. 建立 Cursor 可拆分执行的 Spec 文档包。
```

## 4. v1.4 不做范围

```text id="hb3rsn"
1. 不做完整企业 RBAC。
2. 不做多用户组织管理。
3. 不重写 Web 工作台。
4. 不迁移所有旧 Hermes 配置页。
5. 不做完整 GeneHub 管理。
6. 不做复杂工作流编排器。
7. 不在 Desktop 本地编排专家团队成员。
8. 不让任务窗口直接执行本地文件系统写操作。
9. 不把任务输出伪装成已保存文件，必须明确 outputRef / localPath / remoteRef。
10. 不要求一次性完成 window.work 全局 API，可先用 workApi 包装已有 Preload API。
```

## 11. SSE 事件流方案

### 11.1 标准链路

```text id="e460pa"
TaskComposer submit
  → workApi.task.send()
  → Preload API
  → Main work-task-stream
  → local workspaceChat / nodeskclaw Expert SSE
  → Main event mapper
  → Renderer onWorkTaskEvent
  → WorkTaskEventStore
  → TaskStream 渲染
```

### 11.2 v1.4 兼容策略

v1.4 分两层实现：

```text id="wl2v37"
第一层：Renderer workApi 适配层
  复用已有 window.workspaceChat / window.hermesExperts / window.profileRuntime

第二层：Main work-task-stream 预留层
  新建 src/main/work，但 MVP 可以先只封装 workspace-chat-stream 和 expert SSE
```

### 11.3 事件归一化

```text id="zo02ry"
workspace-chat 原始事件：
  chat.chunk
  chat.tool_progress
  chat.usage
  chat.done
  chat.error
```

## 26. v1.4 最终交付形态

v1.4 完成后，Copilot Desktop 的 Work 工作台应形成如下使用体验：

```text id="ozq779"
用户打开 Work 工作台
  → 看到任务首页
  → 输入一个业务目标
  → 选择专家团队 / 技能 / 应用 / 权限
  → 系统进入任务窗口
  → 中间实时展示 Agent、专家团队、工具调用、审批、输出
  → 底部可持续追问
  → 右侧可查看报告、上下文、参与者、技能、治理
```

工程上形成：

```text id="wfr4po"
src/shared/work
  统一领域契约

src/renderer/src/screens/Work
  Work 工作台 UI

workApi
  Renderer 语义 API

TaskEventStore
  SSE 事件状态中心

TaskStream Blocks
  标准事件块渲染

TaskRightPanel
  输出与上下文工作区
```

这就是 v1.4 的边界：
**先把任务窗口做成稳定、可扩展、可接入实时事件流的主操作平面，再逐步把专家、团队、技能、Web 工作台全部接入任务窗口。**
