# PRD v1.3：WorkBuddy 产品线化与 AI Coding 工程结构优化方案

版本：v1.3
产品线：WorkBuddy
项目：ai-os-desktop / copilot-desktop
模块范围：screen/Hermes、Layout、Main Process Expert MCP、Preload API、Renderer Feature Structure、AI Coding Spec Pack
状态：产品规划 + 工程实施导向 PRD
核心目标：将当前 Hermes 模块从“功能页集合”升级为“WorkBuddy 式专家工作台”，并同步优化项目目录、模块边界、Spec Driver 文档结构，使后续 Cursor / AI Coding 能稳定按模块交付。

---

## 1. 背景

当前第一轮 Expert MCP 调用已经验证通过，用户已能在桌面端完成：

```text
1. 连接 Expert MCP Gateway
2. 浏览远端专家
3. 选择专家技能
4. 召唤专家执行 customer-profiling
5. 查看专家运行记录
6. 查看执行结果
7. 在成果中心预览报告
```

但从产品形态看，当前 Hermes 模块仍然混合了：

```text
1. 业务用户入口
2. 专家召唤入口
3. 本地 Hermes 对话入口
4. MCP 配置入口
5. 技能管理入口
6. 模型与 Provider 配置入口
7. 运行记录与成果入口
```

这会导致两个问题：

```text
1. 对业务用户来说，导航过重，不像 WorkBuddy 的“召唤专家 / 专家团队 / 任务 / 成果”操作链路。
2. 对 AI Coding 来说，页面、组件、API、Feature Hook、类型、Spec 文档边界不够稳定，后续容易继续堆页面。
```

因此 v1.3 的核心不是继续扩展更多专家，而是先完成产品线化与工程结构收敛。

## 3. v1.3 总体目标

### 3.1 产品目标

```text
1. 将 Hermes 模块一级导航收敛为 WorkBuddy 工作流。
2. 将专家、专家团队、任务运行、成果中心作为主流程。
3. 将 MCP、技能、模型、Provider、Memory 等配置能力迁入“高级设置 / 管理区”。
4. 保留本地 Hermes 对话能力，但弱化为“本地助手 / 对话”入口。
5. 提供 WorkBuddy 风格的专家召唤 Drawer、运行详情、成果消费体验。
```

### 3.2 工程目标

```text
1. 不重构 Electron 主壳。
2. 不破坏现有 Main / Preload / Renderer 安全边界。
3. 不让 Renderer 直接访问 nodeskclaw、token、本地文件系统或 MCP endpoint。
4. 将 screen/Hermes 改成 feature-sliced structure。
5. 建立统一 Page Registry、Action Registry、Feature Hook、API Client、Domain Model。
6. 建立 Cursor Spec Pack，后续所有 AI Coding 任务按 Spec 文件推进。
7. 每个功能对象必须有明确输入、输出、状态、错误处理和验收标准。
```

## 4. v1.3 不做范围

```text
1. 不重写 MainPage / WorkspaceOutlet。
2. 不重写 Electron WebContentsView / ShellView。
3. 不重写现有 expert mcp gateway 协议。
4. 不把专家 Profile 安装到 desktop 本地。
5. 不在 desktop 本地编排专家团队成员。
6. 不允许 desktop 传 _routing / _execution / route_config。
7. 不把高级配置功能直接删除，只做导航层级收敛。
8. 不引入新状态库，MVP 继续使用 React Context + hooks。
9. 不引入复杂权限系统，先做 UI 级分组与工程边界。
```

### 9.2 分层规则

```text
shell/
  只做布局与页面装配，不直接调用 API。

registry/
  只定义页面、动作、抽屉元数据，不写业务逻辑。

pages/
  负责页面级编排，可调用 feature hooks，不直接调用 window.* API。

features/
  负责业务数据获取、状态、映射、校验、动作封装。

api/
  只封装 window.hermesExperts / window.hermesAPI / window.mcp API。

model/
  只放类型、枚举、状态机和 schema。

components/
  纯展示组件，不直接调 API，不直接读取全局 Context。

context/
  只保存跨页面共享状态，不承担复杂业务计算。
```

### 9.3 禁止规则

```text
1. 禁止在 pages 中直接调用 window.hermesExperts。
2. 禁止在 components 中直接调用 API。
3. 禁止 Renderer 直接 fetch nodeskclaw。
4. 禁止 Renderer 持有 token。
5. 禁止页面组件内拼 JSON-RPC 原始 payload。
6. 禁止把 run / artifact / expert 类型散落在多个页面。
7. 禁止在 shell 中写业务逻辑。
8. 禁止通过 any 绕过 Preload 类型。
```

## WorkBuddy Product Line

screen/Hermes is the WorkBuddy expert workspace.
Do not treat it as a generic Hermes configuration page.

Primary user flow:
Workbench → Experts / Teams → Summon → Runs → Artifacts.

Renderer rules:
- Pages call feature hooks.
- Feature hooks call api clients.
- Api clients call window.hermesExperts or window.hermesAPI.
- Renderer must not call ipcRenderer, fetch nodeskclaw, read token, or access fs.

Main / Preload rules:
- New backend capability must be implemented in Main first.
- Then expose via ipcMain.handle.
- Then expose via preload.
- Then update index.d.ts.
- Then consume in Renderer.

## 19. 风险与控制

### 19.1 风险：页面重构影响现有调用

控制：

```text
1. 先做 Registry，不改 API。
2. 先保留旧页面路径。
3. 新组件逐步替换旧组件。
4. 保留旧 hook，新增 feature hook 适配。
```

### 19.2 风险：AI Coding 一次修改过大

控制：

```text
1. 每次只执行一个 Spec。
2. 每个 Spec 明确允许修改文件。
3. 每次修改后运行 typecheck。
4. 不允许跨 Main / Preload / Renderer 大范围联动。
```

### 19.3 风险：高级配置入口被业务用户误触

控制：

```text
1. 高级设置默认折叠。
2. 后续增加 advancedMode 开关。
3. MCP / Provider / 模型配置不放主流程区。
```

### 19.4 风险：专家团队编排职责混乱

控制：

```text
1. Desktop 只调用 team tool。
2. 后端负责编排成员。
3. Desktop 只展示 timeline。
4. 不做本地多专家并发调度。
```

## 22. 最终形态

v1.3 完成后，copilot-desktop 的 WorkBuddy 产品线将形成：

```text
WorkBuddy 专家工作台
  ├─ 工作台：状态、推荐、最近任务、最近成果
  ├─ 对话：本地 Hermes 助手
  ├─ 专家：远端 Expert MCP 专家目录
  ├─ 专家团队：后端编排型团队能力
  ├─ 专家运行：任务状态、过程、结果、错误
  ├─ 成果中心：报告、文件、预览、下载、导入
  └─ 高级设置：MCP、技能、模型、Provider、Memory
```

工程上形成：

```text
Renderer:
  shell → registry → pages → features → api → model

Preload:
  typed window APIs

Main:
  domain services + IPC handlers

Spec:
  docs/specs/v1.3-workbuddy-product-line

AI Coding:
  AGENTS.md + .cursor/rules + per-feature spec
```

v1.3 的核心价值是：
把已经跑通的 Expert MCP 能力从“技术调用成功”升级为“可持续迭代的 WorkBuddy 产品线”，同时让 AI Coding 后续开发有稳定的目录、边界、任务粒度和验收标准。
