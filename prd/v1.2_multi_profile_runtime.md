# Hermes Desktop V1.1 下一阶段功能优化与补充方案

建议下一阶段定义为：

```text
Hermes Desktop V1.2
主题：Multi Profile Runtime 工程化增强与可运营化
目标：把 V1.1 从“可运行的多 Profile 架构”升级为“稳定、可观测、可治理、可交付的本地多智能体运行平台”
```

V1.2 不建议继续盲目扩展更多 profile 或复杂业务页面。当前重点应放在 **稳定性、可观测性、权限治理、上下文治理、Web Operator 与 Profile Runtime 的深度集成**。现有 `hermes-desktop` 是单 `BrowserWindow`、Renderer 无 Node.js、Preload 为唯一桥接、Main Process 负责 IPC 与进程管理的架构，因此下一阶段仍应坚持单窗口、多 Workspace 页面、多 Gateway Runtime 的设计边界。

---

# 1. V1.2 总体目标

```text
V1.2 = Profile Runtime 可运营化 + Web Operator 可控自动化 + 多 Profile 协同增强
```

核心补强方向：

```text
1. Multi Profile Gateway 稳定运行
2. Profile Runtime SQLite 管理能力完善
3. Profile Entry 页面体验优化
4. default 主控调度能力增强
5. skill / context / session 共享治理
6. Web Operator profile-aware 深度集成
7. 审计、日志、诊断、恢复能力补齐
8. Windows 桌面部署与私有仓库更新链路增强
```

## P0：必须先做

| 模块                 | 功能                            | 目标                  |

| Plugin Marketplace | skill / capability 插件市场 | 扩展生态           |

# 3. V1.2 推荐功能包

## 3.1 Runtime Reliability Pack

### 目标

让多个 Hermes Gateway 可以长期稳定运行。

### 新增功能

```text
1. profile gateway health polling
2. profile gateway auto restart
3. profile gateway crash detection
4. profile gateway port conflict detection
5. profile gateway startup timeout
6. profile gateway logs viewer
7. profile gateway status recovery after app restart
```

### 新增表字段

`runtime_instances` 增加：

```sql
restart_count INTEGER NOT NULL DEFAULT 0;
last_exit_code INTEGER;
last_crash_at TEXT;
auto_restart INTEGER NOT NULL DEFAULT 1;
health_fail_count INTEGER NOT NULL DEFAULT 0;
```

### 验收标准

```text
1. 手动 kill writer gateway 后，UI 显示 failed。
2. auto_restart=true 时，系统自动重启 writer。
3. 端口被其它进程占用时，返回 PROFILE_PORT_CONFLICT。
4. App 重启后，可恢复上一次 runtime 状态。
5. 每个 profile 可查看 stdout/stderr 日志。
```

## 3.2 Runtime DB Governance Pack

### 目标

让 `profile-runtime.db` 成为可靠控制面，而不是临时状态库。

### 新增功能

```text
1. schema migration version
2. profile-runtime.db backup
3. profile-runtime.db restore
4. config import diff preview
5. config import rollback
6. profile delete / disable
7. orphan runtime cleanup
```

### 关键需求

```text
FR-DB-001：每次 schema 变更必须有 migration。
FR-DB-002：导入 profile-runtime.yaml 前必须展示 diff。
FR-DB-003：覆盖 profile 前必须备份旧配置。
FR-DB-004：删除 profile 不直接删除文件，先 disable。
FR-DB-005：提供“一键修复 runtime db”诊断入口。
```

## 3.4 Delegation Orchestration Pack

### 目标

让 default 不只是“调用其它 profile”，而是能形成可追踪的任务委派链。

### V1.1 当前能力

```text
default -> specialist profile
specialist -> response
delegation_events 记录
```

### V1.2 增强

```text
1. delegation task id
2. delegation status lifecycle
3. multi-profile parallel delegation
4. result merge panel
5. delegation retry
6. delegation timeout
7. delegation context refs
8. delegation audit timeline
```

### 新增任务状态

```ts
type DelegationTaskStatus =
  | "created"
  | "dispatching"
  | "running"
  | "succeeded"
  | "failed"
  | "timeout"
  | "cancelled";
```

### 示例流程

```text
用户在 Portal 输入：
“帮我生成一份招聘智能体的产品方案和代码实现计划”

default:
  -> research: 搜索资料
  -> recruiters: 招聘业务需求
  -> agenter: 智能体架构
  -> coding: 实现计划
  -> writer: 汇总成文档

Portal:
  展示每个 profile 的执行状态
  汇总最终结果
```

### 新增功能

```text
1. skill version
2. skill checksum
3. skill dependency metadata
4. skill compatibility check
5. skill copy preview
6. skill rollback
7. bundled skill installer
8. profile skill matrix
```

### Skill Matrix 示例

```text
              writer  coding  research  finance  agenter
web-operator     ✓       ✓        ✓        ✓        ✓
spec-driver      -       ✓        -        -        ✓
cashflow         -       -        -        ✓        -
profile-delegate ✓       ✓        ✓        ✓        ✓
```

### 验收标准

```text
1. 复制 skill 前可看到目标 profile 是否已有同名 skill。
2. overwrite=true 时自动备份旧 skill。

4. skill_sync_events 可查看完整历史。
5. UI 可查看每个 profile 安装了哪些 skill。
```

## 3.6 Context Governance Pack

### 目标

让 session context 共享可控、可删除、可复用。

### 新增功能

```text
1. shared context version
2. shared context TTL
3. shared context tags
4. shared context summary regeneration
5. shared context usage history
6. shared context attach to delegation
7. shared context revoke
```

### shared_contexts 表增强

```sql
ALTER TABLE shared_contexts ADD COLUMN version INTEGER DEFAULT 1;
ALTER TABLE shared_contexts ADD COLUMN expires_at TEXT;
ALTER TABLE shared_contexts ADD COLUMN tags_json TEXT;
ALTER TABLE shared_contexts ADD COLUMN used_count INTEGER DEFAULT 0;
ALTER TABLE shared_contexts ADD COLUMN last_used_at TEXT;
```

### 验收标准

```text
1. coding session 可生成 context snapshot。
2. snapshot 可共享给 writer / agenter。
3. delegation 时可选择附带 context。
4. context 被使用后 used_count 增加。
5. 用户可 revoke context。
6. revoked context 不再可被 delegation 使用。
```

## 3.7 Web Operator Profile-Aware Pack

### 目标

把 V1.0 的 Web Operator 变成多 profile 共用的桌面自动化工具能力。

### 新增功能

```text
1. browser action 绑定 sourceProfile
2. browser session partition 按 profile 隔离
3. Web Operator action plan
4. sensitive action confirmation queue
5. DOM snapshot history
6. screenshot history
7. selector recorder
8. web task replay
```

### Profile-aware partition 设计

```text
default:
  persist:aios-web-default

coding:
  persist:aios-web-coding

finance:
  persist:aios-web-finance
```

### Desktop Tool Bridge 请求增强

```ts
interface BrowserToolRequest {
  profileId: string;
  source: "user" | "hermes" | "system";
  toolName: string;
  args: Record<string, unknown>;
  requireConfirm?: boolean;
}
```

### 验收标准

```text
1. default 打开的 Web 页面不污染 finance 的登录态。
2. finance 使用 Web Operator 时，audit_events 记录 profile_id=finance。
3. Hermes 发起 browser.click 前，UI 可显示待确认动作。
4. 用户确认后才执行敏感操作。
5. 每次 browser.screenshot 可关联 profile 和 task id。
```

## 3.9 Security & Policy Pack

### 目标

本地多 profile 进入企业场景前，需要有基本操作治理。

### 新增功能

```text
1. profile operation policy
2. sensitive action policy
3. domain allowlist per profile
4. tool allowlist per profile
5. skill install policy
6. delegation allowlist
7. audit export
```

### policy 示例

```yaml
profiles:
  finance:
    allowedTools:
      - browser.get_state
      - browser.screenshot
      - finance.cashflow
    deniedTools:
      - browser.type.password
      - browser.submit.payment
    allowedDomains:
      - erp.company.com
      - bank.company.com

  coding:
    allowedTools:
      - terminal
      - git
      - browser
    delegationTargets:
      - agenter
      - writer
```

### 验收标准

```text
1. finance 不允许执行未授权 browser action。
2. coding 可调用 agenter，但不能调用 finance，除非授权。
3. policy 变更写入 audit。
4. 被 policy block 的操作必须显示原因。
```

# 4. V1.2 推荐实施顺序

## Phase 1：Runtime 稳定性

```text
1. Gateway health polling
2. Auto restart
3. Port conflict detection
4. Runtime logs
5. Status recovery
```

原因：多 profile 同时运行时，稳定性优先级最高。

---

## Phase 2：SQLite Governance

```text
1. Migration
2. Backup / restore
3. Import diff
4. Runtime db doctor
```

原因：V1.1 引入 SQLite 后，必须先保证元数据可靠。

---

## Phase 3：Profile Entry UX

```text
1. Portal 默认入口优化
2. Specialist workspace 优化
3. 最近 profile / session
4. status badge
```

原因：用户每天使用的是入口和页面，不是底层 runtime。

---

## Phase 4：Delegation Task 化

```text
1. delegation task id
2. status lifecycle
3. multi-profile dispatch
4. result merge
```

原因：default 主控能力要从“问答调用”升级成“任务调度”。

---

## Phase 5：Skill / Context Governance

```text
1. skill matrix
2. skill rollback
3. context version
4. context revoke
5. context usage tracking
```

原因：能力和上下文共享必须可治理，否则 profile 会快速混乱。

---

## Phase 6：Web Operator 深度集成

```text
1. profile-aware browser partition
2. action plan
3. sensitive confirm queue
4. screenshot / DOM snapshot history
5. web task replay
```

原因：Web Operator 是桌面自动化的核心，应当从工具变成 profile runtime 的通用能力。

---

# 5. 不建议 V1.2 立即做的内容

```text
1. 不建议立即做多 BrowserWindow。
2. 不建议立即做远程企业 profile server。
3. 不建议立即做 Docker profile runtime。
4. 不建议把 specialist profile 做成完全不同 UI 系统。
5. 不建议让 profile 之间直接合并 state.db。
6. 不建议把 skill 做成 Electron 插件宿主。
7. 不建议让 Renderer 直接访问 SQLite。
```

这些可以放到 V1.3 或 V2.0。

# 9. 最终建议

V1.2 的主线不要继续扩张业务 profile，而是补齐工程能力：

```text
Runtime 稳定性
  ↓
SQLite 治理
  ↓
Profile Entry 体验
  ↓
Delegation 任务化
  ↓
Skill / Context 治理
  ↓
Web Operator 深度集成
  ↓
Observability + Security + Windows 部署
```

这样 V1.2 完成后，Hermes Desktop 才能从“多 Profile 原型”进入“企业内部可用的 Portal Desktop Runtime”。
