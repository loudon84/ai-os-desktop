以下方案基于当前仓库代码分析后整理，目标是让 Cursor 可以直接拆任务实施。

已确认现状：

`src/main/web-operator-task-session-store.ts` 当前表结构中，`task_id` 是主键，`page_url` 是唯一约束，`session_id` 也是唯一约束；`buildTaskId(pageUrl)` 当前通过 `pageUrl` hash 生成 `task_id`。 

`resolveTaskSession(pageUrl)` 当前按 `page_url` 或 `task_id` 查询，仍以页面地址作为绑定入口。

`upsertTaskSession()` 当前要求调用方传入 `taskId`，并按 `task_id` 做 upsert。

IPC 当前只校验 `pageUrl / taskId / sessionId / pageContext`，没有 `source / requestId`。

共享 contract 当前也没有 `source / requestId` 字段。

# PRD：WebOperator Task Session 绑定键调整

## 1. 版本定义

```text
版本号：v6.3.3_task-to-session-request
模块：WebOperator Task Session Store
范围：
- src/main/web-operator-task-session-store.ts
- src/main/web-operator-task-session-ipc.ts
- src/shared/web-operator/web-operator-task-session-contract.ts
- src/preload/web-operator-task-session-api.ts
- WebOperator / JSSDK 触发链路调用方
```

---

## 2. 背景

当前 `task_session` 使用 `pageUrl` 作为业务唯一维度，导致同一个页面地址下不同 JSSDK 请求会被绑定到同一个 task session。

问题表现：

```text
1. 同一个业务页面可能多次触发不同任务。
2. pageUrl 不能表达一次具体 JSSDK 请求。
3. 不同系统、不同 iframe、不同业务按钮可能共用同一个 pageUrl。
4. pageUrl 作为唯一键会导致任务会话误复用。
5. requestId 已经由 JSSDK 侧具备唯一语义，应作为新绑定依据。
```

调整目标：

```text
task_session 唯一业务键从 pageUrl 调整为 source + requestId。
task_id 不再由 pageUrl 生成。
task_id 统一由 source + requestId 序列化后生成。
```

---

## 3. 目标

### 3.1 数据模型目标

`task_session` 增加字段：

```text
source TEXT NOT NULL
request_id TEXT NOT NULL
```

唯一约束调整为：

```sql
UNIQUE(source, request_id)
```

`page_url` 保留，但只作为页面上下文字段，不再作为唯一约束。

---

### 3.2 task_id 生成目标

新增标准函数：

```ts
buildTaskId(source: string, requestId: string): string
```

生成规则：

```text
task_id = "wot_" + sha256(`${normalize(source)}:${normalize(requestId)}`).slice(0, 32)
```

说明：

```text
1. source trim。
2. requestId trim。
3. 两个字段都不能为空。
4. source + requestId 是业务唯一输入。
5. task_id 是派生值，不允许调用方自由传入。
```

---

### 3.3 调用链目标

原调用方式：

```ts
resolve({ pageUrl })
upsert({ taskId, pageUrl, sessionId, pageContext, skill })
```

新调用方式：

```ts
resolve({ source, requestId, pageUrl? })
upsert({ source, requestId, pageUrl, sessionId, pageContext, skill })
```

其中：

```text
1. source 必填。
2. requestId 必填。
3. pageUrl 仍保留，用于页面展示、上下文恢复和兼容旧数据。
4. taskId 由 Main Process store 内部生成，不再由 Renderer / JSSDK 传入。
```

## 5.2 旧结构问题

当前结构：

```sql

page_url TEXT NOT NULL UNIQUE,
session_id TEXT NOT NULL UNIQUE
```

这里 `page_url TEXT NOT NULL UNIQUE` 必须移除唯一性。SQLite 对删除唯一约束不支持直接 `ALTER TABLE DROP CONSTRAINT`，需要重建表。

### 5.3.2 迁移流程

```text
1. 开启 transaction。
2. 创建 schema_meta。
3. 检查 task_session 是否存在。
4. 检查 task_session 是否已有 source / request_id。
5. 如果没有，执行 v1 -> v2 migration。
6. 创建 task_session_v2。

8. 为旧数据生成兼容 source / request_id。
9. 写入 task_session_v2。
10. DROP 旧 task_session。
11. RENAME task_session_v2 TO task_session。
12. 创建新索引。
13. 写入 schema version = 2。
14. 提交 transaction。
```

```text

3. 保留旧 page_context_json。

5. 如果 old.task_id 仍被外部 remove 使用，需要额外兼容 remove。
```

推荐策略：

```text

旧 taskId 不再保证可删除。
```

如需兼容旧 taskId 删除，则增加：


## 7.1 删除旧 buildTaskId 签名

当前：

```ts
export function buildTaskId(pageUrl: string): string {
  return `wot_${createHash("sha256").update(pageUrl.trim()).digest("hex").slice(0, 32)}`;
}
```

目标：

```ts
function normalizeIdentityPart(value: string, field: "source" | "requestId"): string {
  const normalized = value.trim();
  if (!normalized) {
    throw new Error(`${field} is required`);
  }
  return normalized;
}

export function buildTaskId(source: string, requestId: string): string {
  const normalizedSource = normalizeIdentityPart(source, "source");
  const normalizedRequestId = normalizeIdentityPart(requestId, "requestId");
  const raw = `${normalizedSource}:${normalizedRequestId}`;
  return `wot_${createHash("sha256").update(raw).digest("hex").slice(0, 32)}`;
}
```

```text
5. 旧数据库可自动迁移。
```

# 14. 风险点

## 14.1 SQLite 唯一约束变更风险

`page_url UNIQUE` 不能通过简单 ALTER 删除，必须重建表。

处理：

```text
migration 必须事务执行。
失败时不能丢数据。
```

---

## 14.2 调用端仍传 taskId

旧调用端可能继续传：

```ts
taskId
```

处理：

```text
contract 移除 taskId。
TypeScript 编译阶段暴露所有旧调用点。
```

---

## 14.3 JSSDK 未传 requestId

不允许自动用 pageUrl 顶替 requestId。

处理：

```text
缺 requestId 直接拒绝。
错误信息：requestId is required。
```

---

## 14.4 source 命名不统一

建议固定来源枚举或规范：

```text
crm
erp
oa
web-host-bridge
jssdk
manual
legacy-page-url
```

后续可再收敛为类型：

```ts
type WebOperatorTaskSessionSource = string;
```

本次不强行枚举，避免阻断业务系统接入。

---

# 15. 最终交付结果

完成后，`task_session` 的绑定关系从：

```text
pageUrl -> taskId -> Hermes session
```

调整为：

```text
source + requestId -> taskId -> Hermes session
```

`pageUrl` 降级为上下文字段。

这能支持同一业务页面内多次 JSSDK 任务触发，避免不同请求误复用同一个 Hermes 会话。
