# PRD 2：copilot-desktop v6.7_mcp-write-tools-approval

## 1. 版本信息

版本号：v6.7_mcp-write-tools-approval
项目：copilot-desktop
主题：配合 nodeskclaw 服务端审批的 MCP 写操作授权感知
审批模式：nodeskclaw 服务端审批
Desktop 职责：注册 MCP Gateway、传递上下文、刷新授权状态、展示授权结果
Desktop 非职责：不审批、不二次确认、不直接决定写操作是否执行

---

## 2. 当前源码基线

copilot-desktop 当前已具备：

```text
1. MCP Skill Gateway 本地 Proxy。
2. token 由 Main Process 注入。
3. Hermes profile MCP server 注册。
4. 一键诊断。
5. tools preview。
6. invoke test。
7. structured logs。
8. Hermes restart banner。
9. GeneHub Runtime。
10. GeneHub install worker。
```

v6.7 需要在现有 MCP Gateway Runtime 上补充：

```text
1. 服务端审批状态展示。
2. 工具授权状态刷新。
3. 撤销授权感知。
4. Profile / Desktop 上下文传递。
5. MCP_TOOL_APPROVAL_REQUIRED / GRANT_REVOKED 错误展示。
```

---

## 3. 产品目标

### 3.1 核心目标

```text
1. copilot-desktop 不做本地审批。
2. Hermes Agent 调用 write tool 时，直接由 nodeskclaw 决定是否允许。
3. 如果 nodeskclaw 已授权，Hermes Agent 可直接使用。
4. 如果未授权，Hermes Agent 收到 approval required。
5. 如果授权被撤销，Hermes Agent 后续调用失败。
6. Desktop 页面显示当前服务端授权状态。
7. Desktop 可打开 nodeskclaw Approval Center。
```

---

## 4. 非目标

本版本不做：

```text
1. 不做 Desktop 本地审批弹窗。
2. 不做用户二次确认。
3. 不在 Desktop 保存授权最终决策。
4. 不让 Renderer 直接调用 nodeskclaw MCP。
5. 不把 token 写入 Hermes config.yaml。
6. 不在本地绕过 nodeskclaw 执行 write tool。
7. 不缓存 grant 作为最终鉴权依据。
```

---

## 5. 架构原则

### 5.1 认证与授权边界

```text
Authentication：
  由 copilot-desktop Main Process token provider 负责。

Authorization：
  由 nodeskclaw 服务端 grant policy 负责。

Execution：
  Hermes Agent → Desktop Local Proxy → nodeskclaw MCP Gateway → grant check → tool execution。

Desktop：
  只传递上下文，不做审批决策。
```

---

## 6. MCP Proxy 上下文增强

### 6.1 背景

当前 Hermes profile 的 MCP server 指向：

```text
http://127.0.0.1:<port>/mcp
```

为了让 nodeskclaw 能做更精确授权，需要 Desktop Proxy 注入上下文。

### 6.2 新增 Proxy Headers

Local Proxy 转发到 nodeskclaw 时增加：

```http
Authorization: Bearer <accessToken>
X-NoDeskClaw-Desktop-Device-Id: <desktopDeviceId>
X-NoDeskClaw-Hermes-Profile: <profileName>
X-NoDeskClaw-Client: copilot-desktop
X-NoDeskClaw-MCP-Proxy-Version: v6.7
```

说明：

```text
1. Authorization 仍只在 Main Process 注入。
2. Renderer 不接触 token。
3. desktopDeviceId 不是密钥，但不得由 Renderer 任意传入。
4. profileName 来源于 Hermes profile registration。
```

### 6.3 Profile 识别方案

推荐改造 Hermes profile 注册 URL：

```text
http://127.0.0.1:<port>/mcp?profile=<profileName>
```

Local Proxy 解析 query：

```text
profile=default
profile=writer
profile=finance
```

再转发为：

```http
X-NoDeskClaw-Hermes-Profile: default
```

兼容规则：

```text
1. 老 URL 无 profile query 时，默认 profile=default。
2. 新注册写入 profile-scoped URL。
3. listProfileRegistrations 需要兼容旧 URL 和新 URL。
4. 如果 URL 变化，需要提示 Hermes Gateway restart required。
```

## 9. Invoke Test 行为调整

### 9.1 当前逻辑

当前 invoke test 只允许 read tool。

### 9.2 v6.7 调整

在服务端审批模式下，Desktop 本身不审批，因此 invoke test 可以调用 write/admin tool，但必须满足：

```text
1. 用户显式点击 Invoke Test。
2. 请求仍经 Local Proxy。
3. nodeskclaw 决定是否执行。
4. 如果未授权，返回 approval required。
5. 如果已授权，直接执行。
```

安全提示：

```text
Invoke Test 对 write/admin tool 可能触发真实操作。
该操作是否允许由 nodeskclaw 授权策略决定。
```

可选策略：

```text
P0：
  Invoke Test 页面默认隐藏 write/admin tool 执行按钮，只展示“授权状态”。

P1：
  允许管理员模式下执行 write/admin invoke test。
```

建议 P0 不开放 UI invoke write/admin，只让 Hermes Agent 调用真实链路。
Desktop UI 只显示授权状态。

## 11. 撤销授权感知

当 nodeskclaw 撤销 grant 后：

```text
1. Desktop 下次 tools/list 刷新看到 grantStatus=revoked。
2. Tools Preview 标记已撤销。
3. Hermes Agent 后续 tools/call 返回 MCP_TOOL_GRANT_REVOKED。
4. Proxy structured logs 记录 revoked error。
5. Desktop 不做本地拦截。
```

刷新方式：

```text
1. 手动刷新。
2. tools cache TTL 到期刷新。
3. invoke/testRemoteMcp 后刷新。
4. 可选 heartbeat 同步。
```

## 18. Cursor 执行提示词

目标：
