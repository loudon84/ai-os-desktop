下面按 **v6.3.4_weboperator_hermes_host_form_fill** 定义实施方案。核心调整：不要让 Hermes Chat 直接去调用网页，也不要依赖 `callbackURL`；应由 WebOperator Renderer 接收 Hermes 结构化结果后，通过现有 HostBridge `runCommand()` 向当前页面发送 `desktop.host.form.fill`。

当前 Hermes Panel 本质是 WebOperator 侧栏聊天面板，发送消息链路是 `window.hermesDefaultChat.sendMessage(...)`，并不是 HostBridge 执行器；现有资料也说明 Panel 当前只是把 `task.skill` 写入 prompt，由 Hermes runtime 理解执行，不负责本地 skills 或页面动作调用。
同时 Electron 架构要求 Renderer 不能直接访问 Node，本地能力必须经 Preload / IPC 或已有 Renderer Bridge 能力封装，`src/preload/index.ts` 是 Renderer 能力暴露边界。

## 2. 当前问题

截图中 Hermes 已提取出商品参数，但当前回复里出现：

```text
callbackURL 为空，无法直接通过 host.form.fill 回填 CRM 表单
```

这个判断方向不正确。

`host.form.fill` 在当前桌面系统里不是 HTTP callback，而是 WebOperator 内部通过 HostBridge 向当前 Web 页面下发 command。参考你给出的 `HostBridgePanel.tsx`：

```ts
await runCommand({
  commandId: `cmd_test_${Date.now()}`,
  type: "desktop.host.form.fill",
  formType: lastEvent?.formType ?? "product",
  action: "create",
  payload: {
    fields: DEMO_FILL_FIELDS,
    subTables: {
      suppliers: [...]
    },
  },
  createdAt: new Date().toISOString(),
  expectAck: true,
  timeoutMs: 12000,
});
```

因此正确链路应为：

```text
Hermes Chat 输出结构化商品数据
  ↓
WebOperator Hermes Panel 解析结构化数据
  ↓
用户点击“写回表单”
  ↓
调用 HostBridge runCommand()
  ↓
当前 Web 页面收到 desktop.host.form.fill
  ↓
页面执行表单填充并返回 ack
```

```text
Web 页面 HostBridge
  ├─ 上报 host.ready / host.form.fill.applied / ack
  ↓
```

# 13. 关键结论

本功能不要做成：

```text
Hermes Agent -> callbackURL -> CRM
```

应做成：

```text
Hermes Chat 结构化输出
  -> WebOperator Hermes Panel 解析
  -> 用户确认
  -> HostBridge runCommand
  -> desktop.host.form.fill
  -> Web 表单写回
```

这样能复用现有 HostBridge 能力，保持 Electron Renderer / Preload / Main 的边界，并避免把页面操作能力暴露给 Hermes runtime。
