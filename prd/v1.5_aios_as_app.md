# Hermes Desktop v1.5：AI-OS Desktop 集成方案与需求

## 0. 定位结论

`hermes-desktop v1.4` 已完成 Chatbox 风格主 Layout 与组件优化。下一阶段不应把 `ai-os-full` 直接搬进 Electron Renderer，而应把它作为 **本地 AI-OS 全栈 Runtime**，由 Electron Main Process 统一安装、启动、健康检查、状态持久化，并通过主工作区 WebContentsView 挂载。

最终形态：

```text
Hermes Desktop
  = Electron 本地桌面壳
  + Hermes Agent 安装 / 配置 / Gateway Supervisor
  + AI-OS Full 本地 Runtime Supervisor
  + AI-OS 首页入口
  + Web Operator
  + Runtime 状态守卫
```

现有 hermes-desktop 本质是 Electron Shell，智能能力仍委托给本地或远程 `hermes-agent`，桌面侧负责窗口、IPC、文件系统、进程管理和 React UI。

---

## 1. ai-os-full 源码结构判断

`ai-os-full` 是一个 pnpm monorepo，根 `package.json` 已定义 `dev`、`dev:web`、`dev:server`、`build`、`db:migrate` 等脚本，且 workspace 包含 `frontend`、`backend`、`packages/*`。 

### 1.1 前端

前端包名是 `@portal/web`，使用 Next.js 14、React 18、CopilotKit、TanStack Query、TipTap、React Email、FullCalendar、ReactFlow、Recharts、Zustand 等大量业务 UI 依赖。

`frontend/app/[lang]/layout.tsx` 中已经接入：

```text
AuthProvider
TanstackProvider
Providers
GlobalCopilotProvider
DirectionProvider
GlobalCopilotSidebar
```

这说明 `ai-os-full` 前端不是普通静态 SPA，而是 Next App Router + Provider + Route Handler + Middleware 结构。

### 1.2 后端

后端包名是 `@portal/server`，使用 Express 5、Drizzle ORM、PostgreSQL、JWT、Nodemailer、IMAP、S3/MinIO 等能力。

`backend/src/app.ts` 已挂载：

```text
/health
/api/v1/auth
/api/v1/documents
/api/v1/email
/api/v1/workspaces
/api/v1/permissions
/api/v1/users
/api/v1/audit
```

并通过 `authV2Middleware` 做认证与上下文注入。

### 1.3 Hermes 集成能力

`ai-os-full` 前端已有 Hermes BFF：

```text
/api/hermes/health
/api/hermes/health-detailed
/api/hermes/models
/api/hermes/runtime/*
```

`fetchFromGateway()` 会从服务端读取 `HERMES_GATEWAY_BASE_URL`、`HERMES_GATEWAY_TOKEN`，默认指向 `http://localhost:8642`，并做 timeout、Zod 校验、错误归一化。 

`/api/hermes/runtime/[...path]` 还会把 sessions、chat、approval、file、workspace、projects 等路径反代到 `hermes-webui`，默认地址是 `http://localhost:8787`。  

### 1.4 关键判断

`ai-os-full` 不能按“静态页面”方式直接塞进 Electron Renderer。原因：

1. 它依赖 Next Middleware、Route Handler、本地 BFF。
2. 它依赖 Express Backend。
3. 它依赖 PostgreSQL / S3 / Email / Auth 配置。
4. 它已有 Hermes Gateway BFF 和 Hermes WebUI Runtime 代理。
5. 它的 `prepare-frontend-dist.sh` 只是把 `.next` 复制到 `backend/public-frontend`，但当前 `backend/src/app.ts` 没有实际挂载 Next 静态或 standalone server。 

所以首版落地方案必须是：

```text
Electron Main Process 管理 ai-os-full 本地服务
Electron Renderer 只显示主 Layout 与控制状态
AI-OS 页面通过 WebContentsView 加载本地 Next URL
```

## 3. 启动与路由状态机

### 3.1 首次安装阶段保持原逻辑

首次安装不进入 AI-OS 首页，仍进入原 hermes-agent 设置流程：

```text
app boot
  ↓
checkInstall()
  ↓
installed=false
  ↓
Hermes Setup Mode
  ↓
install hermes-agent
  ↓
configure model / api key / gateway
```

现有 `hermes-desktop` 已有 `checkInstall`、`verify-install`、`start-install` IPC 链路。

### 3.2 初始化完成后默认进入 AI-OS 首页

```text
app boot
  ↓
checkInstall()
  ↓
installed=true && configured=true
  ↓
check Hermes Gateway runtime status
  ↓
running=true
  ↓
check AI-OS Runtime status
  ↓
running=true
  ↓
enter AI-OS Desktop Home
```

默认首页不再是单纯 Chat 页面，也不再把 AI-OS 当作 Sidebar 的 WebOperator 子入口，而是：

```text
Default Route = AI-OS Home
```

### 3.3 Hermes Gateway 未运行时强制引导

```text
app boot / user click AI-OS
  ↓
Hermes Gateway status != running
  ↓
show Runtime Guard
  ↓
primary action: Start Hermes Gateway
  ↓
secondary action: Open Hermes Agent Setup Panel
```

规则：

```text
1. Hermes Gateway 未安装：进入安装页
2. Hermes Gateway 未配置：进入配置页
3. Hermes Gateway 已配置但未启动：进入 Runtime Guard
4. Hermes Gateway running：允许进入 AI-OS Home
5. AI-OS Runtime 未启动：显示 AI-OS Runtime 启动面板
```

## 4. AI-OS Runtime 集成方式

### 4.1 默认采用本地服务模式

```text
%LOCALAPPDATA%\HermesDesktop\
  app/
  hermes-agent/
  ai-os-full/
  runtime/
    node/
    pnpm/
    postgres/
  logs/
    hermes-gateway.log
    ai-os-backend.log
    ai-os-frontend.log
    ai-os-runtime.log
  data/
    desktop-runtime.db
```

`ai-os-full` 安装来源支持两种：

```text
A. 安装包内置 ai-os-full dist seed
B. 首次初始化时从私有 Git 仓库 clone / pull
```

生产内部分发推荐：

```text
安装包只内置 bootstrap runtime
ai-os-full 源码从私有 Git 拉取
下载结果落到 %LOCALAPPDATA%\HermesDesktop\ai-os-full
```

### 4.2 启动脚本复用策略

`ai-os-full/scripts/dev-runner.ts` 已经能启动 backend 和 frontend，并写入 `.portal/dev-status/dev-server.json`。它默认使用 `BACKEND_PORT=8000`、`FRONTEND_PORT=3000`，并启动 `@portal/server` 与 `@portal/web`。

但 Desktop 生产版不能直接依赖 dev-runner，需要拆成 Electron Main 管理的进程：

```text
AiOsRuntimeSupervisor
  ├─ runDbMigrate()
  ├─ startBackend()
  ├─ waitBackendHealth()
  ├─ startFrontend()
  ├─ waitFrontendReady()
  └─ persistRuntimeState()
```

开发阶段可以临时复用：

```bash
pnpm --dir "%LOCALAPPDATA%\HermesDesktop\ai-os-full" dev
```

生产阶段改为：

```bash
pnpm --dir "<ai-os-full>" --filter @portal/db migrate
pnpm --dir "<ai-os-full>" --filter @portal/server start
pnpm --dir "<ai-os-full>" --filter @portal/web start
```

## 5. 环境变量注入

Desktop 需要为 `ai-os-full` 写入 `.env.desktop.local` 或直接生成 `.env`：

```env
# AI-OS Backend
PORT=8000
DATABASE_URL=postgresql://postgres:postgres@127.0.0.1:55432/portal
NODE_ENV=production

DEFAULT_TENANT_ID=00000000-0000-0000-0000-000000000001
DEFAULT_WORKSPACE_ID=00000000-0000-0000-0000-000000000001
DEFAULT_USER_ID=00000000-0000-0000-0000-000000000001

JWT_SECRET=<desktop-generated-32-plus-chars>
JWT_REFRESH_SECRET=<desktop-generated-32-plus-chars>

EMAIL_CREDENTIAL_ENCRYPTION_KEY=<desktop-generated-64-hex>

# AI-OS Frontend
NEXT_PUBLIC_SITE_URL=http://127.0.0.1:3000
NEXT_PUBLIC_API_URL=http://127.0.0.1:8000/api/v1
NEXT_PUBLIC_AI_URL=http://127.0.0.1:8001/api/v1

# Hermes Gateway

HERMES_WEBUI_BASE_URL=http://127.0.0.1:8787
```

`ai-os-full` 后端配置明确要求 `DATABASE_URL`、`JWT_SECRET`、`JWT_REFRESH_SECRET` 等关键变量，生产环境还要求 `EMAIL_CREDENTIAL_ENCRYPTION_KEY`。

## 8. Runtime 状态持久化

新增 SQLite 表：

```sql
CREATE TABLE runtime_services (
  service_id TEXT PRIMARY KEY,
  service_type TEXT NOT NULL,
  display_name TEXT NOT NULL,
  status TEXT NOT NULL,
  pid INTEGER,
  port INTEGER,
  url TEXT,
  install_path TEXT,
  started_at TEXT,
  stopped_at TEXT,
  last_health_at TEXT,
  last_error TEXT,
  restart_count INTEGER NOT NULL DEFAULT 0,
  updated_at TEXT NOT NULL
);

CREATE TABLE runtime_events (
  id TEXT PRIMARY KEY,
  service_id TEXT NOT NULL,
  event_type TEXT NOT NULL,
  level TEXT NOT NULL,
  message TEXT NOT NULL,
  payload_json TEXT,
  created_at TEXT NOT NULL
);

CREATE TABLE runtime_settings (
  key TEXT PRIMARY KEY,
  value_json TEXT NOT NULL,
  updated_at TEXT NOT NULL
);
```

服务状态枚举：

```ts
type RuntimeServiceStatus =
  | "unknown"
  | "not_installed"
  | "installed"
  | "configuring"
  | "starting"
  | "running"
  | "degraded"
  | "stopping"
  | "stopped"
  | "error";
```

注意：启动时不能信任上次保存的 PID。必须重新执行：

```text
1. port check
2. health endpoint check
3. process ownership check
4. status reconcile
```

## 9. WebContentsView 挂载策略

AI-OS Web App 使用独立 `WebContentsView`，不要用 iframe。

原因：

```text
1. 需要更好的页面隔离
2. 需要后续接入 BrowserController
3. 需要支持 reload、devtools、session partition
4. 与 WebOperator 架构一致
5. 避免 iframe 与 Next Auth / Cookie / CSP 冲突
```

Main Process 控制：

```ts
aios:view:create
aios:view:attach
aios:view:set-bounds
aios:view:load-home
aios:view:hide
aios:view:destroy
aios:view:reload
```

Renderer 只负责上报容器坐标：

```ts
const rect = containerRef.current.getBoundingClientRect();

window.aiosAPI.setViewBounds({
  x: rect.x,
  y: rect.y,
  width: rect.width,
  height: rect.height,
});
```

## 10. NSIS / 打包配置补充

当前 `electron-builder.yml` 使用 `nsis.oneClick: true`、`perMachine: false`，这适合快速 per-user 安装，但不适合标准 Windows 安装向导。

v1.5 需要调整：

```yaml
nsis:
  oneClick: false
  perMachine: false
  allowToChangeInstallationDirectory: true
  createDesktopShortcut: always
  createStartMenuShortcut: true
  shortcutName: AI-OS Desktop
```

新增资源：

```yaml
extraResources:
  - from: resources/runtime
    to: runtime
  - from: resources/bootstrap
    to: bootstrap
  - from: resources/ai-os
    to: ai-os
asarUnpack:
  - resources/**
  - runtime/**
```

保留现有 electron-vite / electron-builder 链路。当前项目脚本已经具备 `build:win`、`electron-vite build`、`electron-builder --win`。

## 15. 需要确认的工程决策

有 3 个分歧点需要在进入代码实施前确定：

| 决策点             | 方案 A                   | 方案 B                | 默认建议        |
| --------------- | ---------------------- | ------------------- | ----------- |
| ai-os-full 获取方式 | 安装包内置 dist             | 私有 Git clone / pull | B           |
| PostgreSQL      | 本地 portable PostgreSQL | 公司内网 PostgreSQL     | 先 B，后续补 A   |
| AI-OS 前端服务      | Next 本地服务              | 静态导出 / Express 托管   | 先 Next 本地服务 |

当前方案默认按：

```text
私有 Git 拉取 ai-os-full
公司内网或本机 PostgreSQL
Next 本地服务 + WebContentsView 挂载
```

这个路径改动最小，能最大程度复用 `ai-os-full` 已有 Next、BFF、Express、Hermes Gateway 集成。
