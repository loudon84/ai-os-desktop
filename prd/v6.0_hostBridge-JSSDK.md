# copilot-desktop v6.0：HostBridge JSSDK 标准化与 WebOperator 多页签 PRD

## 1. 版本定义

版本号：`v6.0`

基础版本：`v5.7.12`

主题：`HostBridge JSSDK 标准化 + callbackUrl 多页签打开 + Handoff 表单回填`

目标：

1. 将现有 `CrmBridge` 命名统一升级为 `HostBridge`。
2. 将 CRM JSSDK 升级为通用 Host JSSDK，不限定 CRM 场景。
3. 标准化页面向 copilot-desktop 提交的字段协议。
4. 支持 `callbackUrl` 在 WebOperator 中以多页签方式打开。
5. 当 `action=create/edit` 时，在 `callbackUrl` 页面加载完成后执行表单填充。
6. 将可接入 HostBridge 的站点配置迁移到 smc-copilot 安装后的用户可编辑目录。
7. 保留 v5.7.12 旧协议兼容，不影响当前 demo 验证链路。

## 3. 标准事件协议

### 3.1 HostBridge 标准提交事件

JSSDK 提交事件统一为：

```ts
export interface HostBridgeSubmitEvent {
  source: "host-web";
  protocolVersion: "6.0";
  sdkVersion: string;
  requestId: string;
  type: "host.bridge.submit";

  formType: string;
  action: "create" | "edit" | "view" | "analytic";
  callbackUrl?: string;
  skillName?: string;

  trigger: {
    type: "user-click";
    elementId?: string;
    label?: string;
    timestamp: string;
  };

  pageContext: HostBridgePageContext;
}
```

### 3.2 标准字段说明

| 字段            |               必填 | 说明                                                      |
| ------------- | ---------------: | ------------------------------------------------------- |
| `formType`    |                是 | 单据类型，例如 `product`、`customer`、`quote`、`order`、`supplier` |
| `action`      |                是 | 当前动作：`create`、`edit`、`view`、`analytic`                  |
| `callbackUrl` | `create/edit` 必填 | Desktop 处理完成后需要打开的业务页面                                  |
| `skillName`   |                否 | 指定 Hermes skill；为空时走配置路由默认 skill                        |
| `pageContext` |                是 | 当前页面上下文                                                 |
| `trigger`     |                是 | 必须来自用户点击                                                |

### 3.3 action 行为

| action     | 行为       | 是否打开 callbackUrl | 是否 fillForm |
| ---------- | -------- | ---------------: | ----------: |
| `create`   | 生成新建单据草稿 |                是 |           是 |
| `edit`     | 生成编辑单据补丁 |                是 |           是 |
| `view`     | 查看上下文    |                否 |           否 |
| `analytic` | 分析当前单据   |                否 |           否 |

### 3.4 pageContext 结构

```ts
export interface HostBridgePageContext {
  app: string;
  url: string;
  title?: string;

  entityType?: string;
  entityId?: string;
  entityName?: string;

  fields?: Record<string, string | number | boolean | null>;
  selectedIds?: string[];
  safeText?: string;

  data?: Record<string, unknown>;
}
```

示例：

## 5. callbackUrl 多页签打开能力

### 5.1 目标

当 HostBridge 事件包含 `callbackUrl`，copilot-desktop 不应覆盖当前 WebOperator 页面，而应在 WebOperator 中以多页签方式打开。

场景：

```text
商品查看页
  ↓ 点击 AI 生成新建草稿
HostBridge submit:
  formType=product
  action=create
  callbackUrl=http://localhost:3000/product-form.html?action=create
  skillName=crm-product-create
  ↓
Desktop 调用 skill
  ↓
生成 fillFormPayload
  ↓
WebOperator 新增页签打开 callbackUrl
  ↓
callbackUrl 页面 ready
  ↓
Desktop 下发 desktop.host.form.fill
```

### 5.2 WebOperator 多页签模型

新增 tab 数据结构：

```ts
export interface WebOperatorTab {
  tabId: string;
  title: string;
  url: string;
  status: "loading" | "ready" | "failed";
  kind: "normal" | "host-callback";
  webContentsId?: number;
  hostBridge?: {
    requestId?: string;
    formType?: string;
    action?: "create" | "edit" | "view" | "analytic";
    callbackUrl?: string;
    handoffId?: string;
  };
  createdAt: string;
  updatedAt: string;
}
```

### 5.3 Main Process tab 管理

新增模块：

```text
src/main/browser/web-operator-tabs.ts
```

职责：

1. 创建 WebOperator tab。
2. 关闭 WebOperator tab。
3. 激活 WebOperator tab。
4. 根据 tabId 获取 webContents。
5. 根据 callbackUrl 查找 tab。
6. 将 HostBridge handoff 与 tabId 绑定。

新增 IPC：

```text
web-operator-tabs:list
web-operator-tabs:create
web-operator-tabs:activate
web-operator-tabs:close
web-operator-tabs:get-active
web-operator-tabs:open-callback
```

### 5.4 callbackUrl 打开规则

函数：

```ts
openHostCallbackTab(input: {
  requestId: string;
  formType: string;
  action: "create" | "edit";
  callbackUrl: string;
  handoffId: string;
}): Promise<WebOperatorTab>
```

规则：

1. `callbackUrl` origin 必须在 `bridge-config.json` 允许站点中。
2. 优先复用相同 `requestId + callbackUrl` 的 tab。
3. 不复用普通 tab。
4. 不覆盖当前商品查看页 tab。
5. 新 tab 的 `kind` 设置为 `host-callback`。
6. 新 tab 的 `hostBridge.requestId` 绑定当前 requestId。
7. callback tab 加载完成后等待 `host.page.ready`。
8. 页面 ready 后由 handoff orchestrator 下发 fill command。

### 5.5 Renderer UI

WebOperator 顶部增加 tab bar：

```text
[商品查看] [Local Hermes] [商品新建 *]
```

tab 展示规则：

1. 普通页面显示标题。
2. HostBridge callback 页签显示业务动作：

   * `product:create`
   * `product:edit`
3. loading 状态显示 loading icon。
4. handoff pending 显示小圆点。
5. fillForm delivered 后小圆点变为成功状态。
6. failed 后显示错误标记。

## 6. Host Handoff 流程

### 6.1 Handoff Record

```ts
export interface HostHandoffRecord {
  handoffId: string;
  requestId: string;
  tabId?: string;

  formType: string;
  action: "create" | "edit";
  callbackUrl: string;
  callbackOrigin: string;
  skillName?: string;

  sourceEvent: HostBridgeSubmitEvent;

  fillFormPayload: {
    fields: Record<string, unknown>;
    subTables?: Record<string, Array<Record<string, unknown>>>;
    meta?: Record<string, unknown>;
  };

  status:
    | "pending"
    | "callback-opening"
    | "callback-loaded"
    | "ready-received"
    | "delivering"
    | "delivered"
    | "failed"
    | "expired";

  commandId?: string;
  ack?: HostDesktopCommandAck;

  createdAt: string;
  updatedAt: string;
  expiresAt: string;
}
```

### 6.2 create/edit 标准流程

```text
host.bridge.submit
  ↓
validateHostBridgeEvent
  ↓
routeHostBridgeEvent
  ↓
runHostSkill
  ↓
create HostHandoffRecord
  ↓
openHostCallbackTab(callbackUrl)
  ↓
callbackUrl loaded
  ↓
host.page.ready
  ↓
match handoff
  ↓
send desktop.host.form.fill
  ↓
wait ack
  ↓
handoff status = delivered
```

### 6.3 ready 事件

页面加载后由 JSSDK 主动发送：

```ts
export interface HostPageReadyEvent {
  source: "host-web";
  protocolVersion: "6.0";
  sdkVersion: string;
  requestId: string;
  type: "host.page.ready";

  formType?: string;
  action?: "create" | "edit" | "view" | "analytic";

  pageContext: {
    app: string;
    url: string;
    title?: string;
    entityType?: string;
    entityId?: string;
    entityName?: string;
  };
}
```

说明：

1. `host.page.ready` 不要求用户点击。
2. `host.page.ready` 必须校验 origin。
3. `host.page.ready` 用于匹配 pending handoff。
4. 匹配成功后才能下发表单填充 command。

```ts
contextBridge.exposeInMainWorld("CopilotDesktopCRM", legacyAdapter);

```

### 9.2 postMessage channel

新 channel：

```text
host.bridge.submit
host.page.ready
host.desktop.command
host.desktop.command.result
```

兼容旧 channel：

```text
crm.desktop.bridge
crm.desktop.command
crm.desktop.command.result
```

### 9.3 preload 行为

1. 捕获用户真实点击时间。
2. `submit()` 必须校验最近用户点击。
3. `ready()` 不要求用户点击。
4. `submit()` 调用 `ipcRenderer.invoke("host-bridge:emit", event)`。
5. `ready()` 调用 `ipcRenderer.invoke("host-bridge:page-ready", event)`。
6. Main 下发 command 时，preload 转发到页面：

```js
window.postMessage({
  source: "copilot-desktop",
  channel: "host.desktop.command",
  protocolVersion: "6.0",
  command
}, window.location.origin);
```

7. 页面 ack 时，preload 转发：

```ts
ipcRenderer.invoke("host-bridge:command-result", ack)
```


匹配顺序：

1. `formType + action + skillName`
2. `formType + action + *`
3. `formType + action`
4. legacy event type
5. fallback route
6. 未匹配返回 `ROUTE_NOT_FOUND`

## 12. 安全要求

1. JSSDK 不允许访问 Node.js。
2. 页面不允许直接访问 `ipcRenderer`。
3. Preload 只暴露 `CopilotHostBridge`。
4. `submit()` 必须要求真实用户点击。
5. `ready()` 不要求用户点击，但必须校验 origin。
6. `callbackUrl` origin 必须在 allowedOrigins。
7. `skillName` 必须在 allowedSkills。
8. `formType` 必须在 allowedFormTypes。
9. `action` 必须在 allowedActions。
10. `pageContext` 必须过滤敏感字段。
11. Desktop 下发 command 必须包含 commandId。
12. command ack 必须记录。
13. 非法事件全部写入 audit log。
14. WebOperator callback tab 不允许打开非白名单 URL。
15. legacy alias 只做兼容，不写入新标准文档为主接口。

## 16. 验收标准

### 16.1 命名验收

1. 新增类型不再使用 `CrmBridge`。
2. 新增 IPC 使用 `host-bridge:*`。
3. 新增 command 使用 `desktop.host.*`。
4. UI 文案从 `CRM Context` 改为 `Host Context`。
5. legacy `crm-*` 仍可运行。

### 16.2 配置验收

1. 首次启动生成：

```text
%APPDATA%/smc-copilot/bridge-config.json
```

2. WebOperator 可显示当前配置路径。
3. 修改 allowedOrigins 后 reload 生效。
4. 配置错误不导致应用崩溃。

### 16.3 callbackUrl 多页签验收

1. WebOperator 当前打开商品查看页。
2. 点击 `AI 生成新建草稿`。
3. Desktop 不覆盖商品查看页。
4. WebOperator 新增 tab 打开 `/product-form.html?action=create`。
5. tab 标记为 `product:create`。
6. callback 页面 ready 后收到 fillForm。
7. handoff 状态变为 `delivered`。

### 16.4 create 回填验收

1. `action=create` 提交。
2. Main 创建 handoff。
3. callbackUrl 新 tab 打开。
4. `host.page.ready` 被接收。
5. Main 下发 `desktop.host.form.fill`。
6. 商品主表字段被填充。
7. 供应商子表被填充。
8. 页面回传 ack。
9. WebOperator 显示 ack 成功。

### 16.5 edit 回填验收

1. `action=edit` 提交。
2. callbackUrl 为当前商品编辑页。
3. 新 tab 打开编辑页。
4. 页面 ready。
5. Main 下发表单 patch/fill。
6. 页面填充字段但不自动保存。
7. ack 成功。

### 16.6 analytic 验收

1. `action=analytic` 提交。
2. 不打开 callbackUrl。
3. 打开 Host Context panel 或 Hermes skill panel。
4. 展示 pageContext。
5. 显示 skillName。

### 16.7 安全验收

1. 非 allowedOrigins 被拒绝。
2. 非用户点击 submit 被拒绝。
3. `callbackUrl` 非白名单 origin 被拒绝。
4. 非 allowedSkills 被拒绝。
5. payload 包含 token/password/cookie 被拒绝。
6. 所有拒绝记录 audit log。
