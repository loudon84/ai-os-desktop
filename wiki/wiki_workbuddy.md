基于详尽的调查,我现在来绘制 WorkBuddy 的产品设计结构全景图。我会用多个视角来呈现:
WorkBuddy 的核心是 **Agent 智能体循环** — 这是它与普通聊天机器人的根本区别。让我把这个循环拆开来看:
除了核心架构,WorkBuddy 最精妙的设计在于它的**可扩展生态**。四种扩展机制各有定位、互不重叠:
---

## WorkBuddy 产品设计结构完整分析

### 一、整体设计哲学

WorkBuddy 不是传统的聊天机器人，而是一个 **"自主智能体操作系统"**。它的核心设计理念是：

> **给 AI 一个身体，而不是只给它一张嘴。**

它通过 **Agent 循环** 让 AI 能"动手"——读写文件、执行代码、搜索网络、操控应用——而不仅仅是对话。这套系统建立在三层记忆和四个可扩展维度的基础之上。

---

### 二、七层架构 (从外到内)

| 层次 | 名称 | 核心职责 |
|------|------|----------|
| L1 | **内容安全与合规层** | 在最外层拦截政治敏感、色情、隐私侵犯等内容，确保所有输出合规 |
| L2 | **三种工作模式** | Craft(即时执行) / Plan(先规划) / Ask(只读问答) — 用户控制 AI 的自主权级别 |
| L3 | **Agent 智能体循环** | 7 步闭环: 分析上下文 → 推理决策 → 选择工具 → 执行动作 → 观察结果 → 迭代/交付 |
| L4 | **能力生态层** | 工具 + Skills + MCP连接器 + 插件 — 四种扩展机制为 Agent 提供手脚 |
| L5 | **记忆与持久化层** | Cloud/User/Workspace 三层记忆，从全局到局部、从自动到显式 |
| L6 | **身份与人格系统** | SOUL.md/IDENTITY.md/USER.md — 让 AI 有"人格"而非纯粹工具 |
| L7 | **自动化引擎** | 定时循环 (RRULE) 和一次性调度 — 让 AI 在没有用户时也能工作 |

---

### 三、Agent 循环 — 核心引擎

这是 WorkBuddy 与 GPT/ChatGPT 等对话式 AI 最根本的区别：

```
用户输入 → [分析上下文] → [推理决策] → [选择工具] → [执行] → [观察结果]
                                        ↑                              |
                                        └──── 任务未完成，迭代循环 ←────┘
                                                          ↓
                                                    任务完成 → [交付成果]
```

**关键设计点:**
1. **每一步都有状态意识** — 不只是"生成文本"，而是知道自己在任务的哪个阶段
2. **工具即行动** — 40+ 个工具覆盖文件、代码、网络、可视化、部署等全部操作
3. **自动迭代** — 不需要用户每步确认，AI 自己判断"做完没有"
4. **成果交付标准化** — `present_files` 统一出口，HTML 自动预览、文件自动展示

---

### 四、四种扩展机制对比 (关键架构决策)

WorkBuddy 设计了 **四个互不重叠的扩展维度**，每种解决不同场景：

| 机制 | 复杂度 | 职责边界 | 类比 |
|------|--------|----------|------|
| **Skills** | 低 (Markdown) | 教 Agent **怎么做**一件事 | VS Code 的扩展设置 |
| **Plugins** | 高 (代码包) | 给 Agent **加一个器官** | Chrome 扩展 |
| **MCP Connectors** | 中 (配置+服务) | 给 Agent **接一条电话线** | Zapier 连接器 |
| **Experts** | 低 (Prompt) | 给 Agent **换一个人格** | GPTs |

**设计精妙之处:** 这四种机制选择了不同的 "抽象层级" 和 "技术入口"，使得任何能力扩充都有最合适的路径，不会出现"杀鸡用牛刀"的问题。

---

### 五、三层记忆架构

```
Cloud Memory (云端，自动)
  └── 服务端根据历史行为生成的 Profile
  └── 会话历史检索 (conversation_search)
      ↓
User Memory (~/.workbuddy/MEMORY.md，用户级，显式)
  └── 跨项目的偏好、习惯、规则
      ↓
Workspace Memory (.workbuddy/memory/，项目级，显式)
  └── 每日日志 (YYYY-MM-DD.md)
  └── 长期项目笔记 (MEMORY.md)
```

**设计目的:** 三层逐级聚焦 —— Cloud Memory 管"我是谁"，User Memory 管"我喜欢什么"，Workspace Memory 管"我们在做什么"。

---

### 六、安全设计亮点

1. **沙箱隔离执行** — 所有 Bash/PowerShell 命令默认在沙箱中运行
2. **内容政策层顶置** — 在 Agent 循环之前做第一道过滤
3. **个人文件操作保护** — Desktop/Downloads/Documents 等高风险目录有专门的 8 条规则守卫
4. **Skill 安全审计** — 安装新 Skill 时自动检查 P0/P1/P2 风险等级
5. **MCP Trust 机制** — 外部连接器需用户显式授权才能激活

---

### 七、值得关注的设计模式

1. **文件即配置 (File-as-Config)** — 所有记忆、身份、设置都用 Markdown/JSON 文件存储，人类可读可编辑
2. **渐进式自主权** — Craft/Plan/Ask 三种模式让用户控制 AI 的行动边界
3. **工具生态的延迟加载** — MCP 工具不直接暴露，通过 ToolSearch + DeferExecute 两段式调用，节省上下文
4. **可视化内联** — Visualizer 将 SVG/HTML 直接注入对话流，结果不离开聊天界面
5. **Skills 的积累-反思-修正闭环** — Skill 不仅是静态指令，还能在每次使用后被自动优化

---

这套架构将 **LLM 推理能力** 与 **操作系统级执行能力** 通过 Agent 循环粘合在一起，外层用安全政策包裹，底层以文件系统为持久化基座。它是一个为"让 AI 真正干活"而设计的完整操作系统，而非"让 AI 聊天"的工具。
